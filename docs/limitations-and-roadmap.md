# 알려진 한계와 로드맵

> 무엇을 의도적으로 안 했고, 무엇이 아직 검증 미완이며, 어디로 갈지를 정직하게 적습니다.
> "다 됐다"가 아니라 **"무엇을 왜 안 했는지 아는 것"** 이 엔지니어링 성숙도라고 봅니다.

## 1. 현재 구현 범위 (MVP) — "좁고 깊게"

이 프로젝트의 비전은 **풀 커머스 마켓플레이스 플랫폼**이지만, 첫 구현은 **돈 한 줄기를 깊게** 파는 데 집중합니다:

```
장바구니 → 주문 → PG결제 → 배송 → 멀티셀러 정산 → 부분환불 → 4-way 대사
```

이 한 줄기가 네 가지 어려운 역량(정산+대사 / 분산 정합성 / AI 테스트 전략 / 동시성·스케일)을 전부 지나가도록 설계했습니다. 넓이보다 깊이를 택한 이유는 [`design/implementation-spec.md`](./design/implementation-spec.md) §0~2를 참조하세요.

## 2. 알려진 기술적 한계 (검증 경계)

설계는 옳게 했으나, **사이드 프로젝트 환경에서 증명 가능한 선**을 정직하게 긋습니다:

| 영역 | 설계 | 검증 한계 |
|---|---|---|
| **대용량 처리량** | 병목→분리 매핑·진화 경로 설계 완료 | 로컬 단일 머신 측정 + 멀티노드는 *설계만*. 진짜 대규모 운영 검증은 못 함 |
| **마감 락 starvation** | 2단계 마감으로 완화 설계 | 고부하에서 마감 지연 분포 *실측*은 부하 테스트(§7) 완료 후 |
| **대사 임계값** | grace/tolerance를 PG SLA 기반으로 설계 | 실제 PG 정산 지연 *실데이터*가 없어 임계값은 가정값 |
| **장애 대응** | 멱등 재실행·정산 미아 fallback 설계 | 실제 인시던트 운영(커뮤니케이션·RCA) 경험은 없음 |

> 이 한계들은 *숨기지 않고* 드러냅니다. 측정·시뮬레이션으로 덮을 수 있는 부분과, 실서비스 운영에서만 채워지는 부분을 구분하는 것 자체가 설계 역량이라고 봅니다.

## 3. 로드맵 — 풀 커머스 플랫폼으로 확장

현재 정산 한 줄기에서, 측정·필요가 정당화하는 순서로 확장합니다:

### 도메인 확장
- [ ] **유저·인증·인가** — 회원, 로그인(현재는 시드/헤더 stub), 셀러/구매자/관리자 권한
- [ ] **상품·카탈로그** — 상품 등록, 옵션, 재고 관리 고도화
- [ ] **쿠폰·할인·포인트** — 정산 부담주체 분해와 연동
- [ ] **교환·반품 물류** — 회수 + 재출고 + 차액결제(현재 부분환불 1종만)
- [ ] **실시간 배송 추적 + 지도** — 배송 이벤트 스트리밍, 위치 시각화

### 인프라·스케일 진화 ([`design/implementation-spec.md`](./design/implementation-spec.md) §7)
- [ ] **부하 실측** — 핫로우 락·outbox lag knee를 그래프로(§7-3b)
- [ ] **첫 서비스 추출** — 측정이 정당화하는 서비스(정산/재고)를 별 프로세스+별 DB로, before/after 실측
- [ ] **이벤트 인프라** — outbox 릴레이 → Kafka, 파티셔닝
- [ ] **CQRS 읽기 모델 / read replica**
- [ ] **관측성** — Prometheus/Grafana로 TPS·p99·락 대기·대사 drift 대시보드

### 검증 강화
- [ ] property test(jqwik) 커버리지 확대
- [ ] 결정론적 시뮬레이션(장애 주입 + 시드 재현)
- [ ] CI에 부하 테스트(k6) 통합

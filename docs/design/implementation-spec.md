> 📎 설계 학습 노트에서 발췌·정리. "README 골격"이던 원본 §6은 실제 [`../../README.md`](../../README.md)로 구현됐습니다.
>
> 설계([`system-design.md`](./system-design.md))는 "무엇이 어떻게 생겼나(DDL·도메인·경계)"였습니다. 이 문서는 **"그중 무엇을, 어디까지, 어떻게 돌아가게 만들고, 어떻게 데모하나"** 입니다.
> 사이드 프로젝트(백엔드 포트폴리오)로 내놓기 위한 **구현 기획서**입니다.

## 0. 결론 한 줄 + 왜 기획부터인가

**한 줄:** 마켓플레이스의 **돈 한 줄기**(장바구니→주문→결제→배송→멀티셀러 정산→부분환불→4-way 대사)를 **end-to-end로 진짜 돌아가게** 만들고, 그 위에서 **분산 정합성·동시성·AI 시대 테스트 전략**을 코드로 증명한다.

**왜 기획(범위)부터인가:**

- AI로 짜면 **경계가 없으면 무한히 그럴듯한 걸 만든다.** 기획서 = AI에게 주는 작업 경계(파일 12 §13-6의 1번).
- 포트폴리오의 함정은 "넓고 얕게". 대기업 백엔드 평가는 **"좁고 깊게"** — 어려운 부분(정산·대사·정합성)을 한 줄기로 끝까지 돌리는 쪽이 점수가 된다.
- 기획서가 박는 것은 딱 셋: **(1) 목표 한 줄, (2) 만들 것/안 만들 것, (3) 보여줄 골든 시나리오.**

## 1. 이 프로젝트가 증명하는 4가지 역량 (헤드라인)

| # | 역량 | 골든 시나리오 어디서 드러나나 | 핵심 산출물 |
|---|---|---|---|
| ① | **멀티셀러 정산 + 4-way 대사** | 2개 셀러 주문 → 셀러별 정산 라인 → 부분환불 음수 라인 → 4경로 대사 일치 | `settlement_line`(부호 amount), 대사 배치 + drift 메트릭 |
| ② | **분산 정합성**(outbox·멱등·이벤트) | 결제확정 → outbox → 배송 도메인이 이벤트로 셀러별 shipment 생성 | outbox 테이블·릴레이, `source_event_id` 멱등, 상태머신 가드 |
| ③ | **AI 시대 테스트 전략**(파일 12 §13) | 위 모든 단계에 property·동시성·결정론 시뮬·대사·DB 불변식 | Testcontainers 실 DB, jqwik property, 시드 시뮬, CHECK/EXCLUDE |
| ④ | **동시성·스케일** | 따닥 결제확정(멱등), 마감 중 라인 유입(FOR SHARE), 정산 배치 | 동시성 테스트, 읽기 모델, 정산 배치 엔지니어링 |

> 4개가 **하나의 한 줄기**를 지나가도록 시나리오를 짰다 — 기능을 늘리지 않고 깊이로 4개를 다 보여주는 게 이 기획의 핵심.

## 2. 범위 — 만들 것 / 안 만들 것 (Scope In / Out)

### ✅ Scope In (이번에 진짜 만든다)

```
[구매자]  장바구니 담기 → 주문 생성(멀티셀러 1주문) → PG 결제 승인/확정
[배송]    결제확정 이벤트 → 셀러별 shipment 생성 → 배송 상태 전이(이벤트 append)
[정산]    배송완료 → settlement_cycle 배치 → settlement_line(수수료·정산액)
[환불]    부분환불 1종(주문 라인 일부 환불) → 환불 settlement_line(음수) → 정산 반영
[대사]    4-way 대사 배치(원장 SUM ↔ PG파일 ↔ 복식부기 ↔ 은행) → drift=0 검증
[안전망]  property·동시성·결정론 시뮬 테스트 + DB 불변식 + 대사 메트릭
```

### ❌ Scope Out (이번엔 안 만든다 — 명시적으로 자른다)

| 자르는 것 | 이유 |
|---|---|
| 쿠폰·할인·포인트 | 정산 한 줄기와 직교, 넓이만 늘림 (파일 06 설계는 있되 구현 보류) |
| 교환(EXCHANGE)·반품 회수 물류 | 환불 1종으로 "돈 흐름"은 이미 증명됨 |
| 실제 프론트엔드 UI | 데모는 Swagger/API + README 시퀀스로 대체 (§6) |
| 회원/인증/권한 | 셀러·구매자는 시드 데이터로 고정, 토큰은 헤더 stub |
| 실 PG·실 택배 연동 | **외부 경계만 mock**(파일 12 §13-3) + 스테이징 1회 contract test |
| 검색·추천·리뷰 | 도메인 밖 |

> **자르는 것을 명시하는 게 기획의 절반.** "안 만든다"가 없으면 AI도 사람도 무한히 늘린다.

## 3. 골든 데모 시나리오 (이 프로젝트의 "한 줄기")

면접·README에서 이 **한 시나리오**를 처음부터 끝까지 돌린다. 단계마다 (a)무슨 일 (b)어떤 API (c)상태변화 (d)걸리는 불변식 (e)증명하는 역량.

```
S0. 시드: 셀러 A·B, 상품 a(A), 상품 b(B), 구매자 U, 수수료정책(A=10%, B=15%)

S1. 장바구니 → 주문        POST /carts/{id}/items, POST /orders
    한 주문에 셀러 A·B 라인 공존(멀티셀러)            → 역량①의 토대
    불변식: 주문 총액 == Σ라인 금액

S2. 결제 승인·확정          POST /payments → (PG mock) → POST /payments/{id}/confirm
    confirm은 멱등(같은 idempotency_key 두 번 → 한 번 효과)  → 역량②④
    confirm 커밋과 같은 TX에 outbox INSERT(dual-write 회피)  → 역량②
    불변식: refunded_amount=0, payment.status 단조 전이

S3. 배송 생성(이벤트)        outbox 릴레이 → "PaymentConfirmed" → 배송 도메인 구독
    셀러별 shipment 2건 생성(A박스·B박스)                  → 역량②
    멱등: 같은 source_event_id 재수신 → 23505 swallow+ack
    불변식: shipment.order_id FK, over-ship 거부

S4. 배송 완료              POST /shipments/{id}/events {DELIVERED}
    배송완료 이벤트 → 정산 가능 표시                      → 역량②

S5. 정산 배치             (스케줄) settlement run
    셀러별 settlement_line 생성: 매출(+), 수수료(−)        → 역량①④
    A: +상품가, −10%; B: +상품가, −15%
    불변식: ck_settline_sign(부호 규칙), 사이클 반열림·겹침금지

S6. 부분환불              POST /orders/{id}/refunds {line, amount}
    (PG mock 환불) → 환불 settlement_line(음수) 추가        → 역량①
    불변식: 0 ≤ 누적환불 ≤ 결제금액 (CHECK), 멱등키 UNIQUE

S7. 4-way 대사 배치        (스케줄) reconciliation run
    ① settlement_line SUM  ② pg_settlement_raw
    ③ 복식부기 CLEARING_PG 잔액=0  ④ 은행 실입금(파일 시드)
    네 숫자 일치 → drift=0 메트릭; 불일치 → break 알람       → 역량①③

S8. (동시성 데모)          따닥 confirm 2스레드, 마감 중 라인 유입
    멱등·FOR SHARE·EXCLUDE가 진짜 방어임을 동시성 테스트로 증명 → 역량③④
```

```
 [U] ──담기/주문──▶ ORDER ──확정──▶ PAYMENT ─(outbox)─▶ SHIPPING
                     │                                    │
                     │                              배송완료 이벤트
                     ▼                                    ▼
                  REFUND ◀──부분환불──             SETTLEMENT(배치)
                     │                                    │
                     └────────────▶ RECON(4-way 대사) ◀───┘
                                    drift == 0 ?
```

## 4. 기능 단위 명세 — "각 기능을 어떻게 구현/데모하나"

말씀하신 **"각자 어떻게 구현됐으며 클릭이든 뭐든 어떻게 동작하나"** 를 기능별로 박는다. UI=Swagger이므로 **"클릭" = 해당 엔드포인트 실행**.

> 표 읽는 법: **데모 동작** = README/면접에서 누를 버튼(=API). **상태변화** = 그 결과 DB가 어떻게 바뀌나. **가드** = 그 기능을 지키는 불변식/테스트.

| 기능 | API(클릭) | 입력 | 상태변화(DB) | 가드(불변식·테스트) |
|---|---|---|---|---|
| 장바구니 담기 | `POST /carts/{id}/items` | 상품·수량 | cart_item upsert | 수량>0 CHECK |
| 주문 생성 | `POST /orders` | cart_id | order + order_line(멀티셀러) | 총액=Σ라인, property: 보존 |
| 결제 승인 | `POST /payments` | order_id, 금액 | payment(PENDING), PG mock 호출 | 금액=주문총액 |
| 결제 확정 | `POST /payments/{id}/confirm` | idempotency_key | payment(PAID) + **outbox INSERT(같은 TX)** | 멱등 UNIQUE, 상태머신, 동시성 테스트 |
| 배송 생성 | (이벤트 구독, API 없음) | PaymentConfirmed | shipment×셀러수 | source_event_id 멱등, FK |
| 배송 이벤트 | `POST /shipments/{id}/events` | 상태코드 | shipment_event append | 상태 전이 가드, append-only |
| 정산 실행 | `POST /admin/settlements/run` | cycle | settlement_line(±) | ck_settline_sign, 사이클 EXCLUDE |
| 부분환불 | `POST /orders/{id}/refunds` | line, amount, key | refund + 음수 settlement_line | 0≤누적≤결제 CHECK, 멱등, property |
| 4-way 대사 | `POST /admin/recon/run` | date | recon_result + drift 메트릭 | drift=0, break 시 알람 |
| 정산 조회 | `GET /sellers/{id}/settlements` | seller_id | (읽기) | 읽기 모델 |

<details>
<summary>🧩 <b>각 기능을 "어떻게 데모로 보여주나" — Swagger 클릭 흐름 (UI 없이 흐름을 눈에 보이게)</b></summary>

UI를 안 만들어도 **흐름이 눈에 보이게** 하는 3가지 장치:

1. **Swagger UI `Try it out`** — 각 엔드포인트를 순서대로 눌러 응답·상태코드 확인. README에 "1번 누르고 2번 누르고…" 순서를 명시.
2. **`.http` 시나리오 파일**(IntelliJ/VS Code REST Client) — 골든 시나리오 S1~S8을 한 파일에 담아 위에서 아래로 실행하면 한 줄기가 재생됨. 변수(`{{orderId}}`)로 단계 연결.
3. **상태 스냅샷 엔드포인트** — `GET /admin/debug/order/{id}` 가 주문·결제·배송·정산·환불·대사를 한 화면 JSON으로 반환 → "지금 시스템이 이 상태"를 한눈에. (데모 전용, 운영 OFF)

> "클릭"의 본질은 **상태가 바뀌는 걸 보여주는 것**. UI 버튼이든 Swagger든 `.http`든, 핵심은 *액션 → 상태변화*를 추적 가능하게 만드는 것이다.

</details>

## 5. 빌드 순서 (마일스톤) — TDD/DDD 순으로

파일 12 §13-6 레시피를 이 프로젝트에 적용한 실제 작업 순서. **불변식·테스트 먼저, 그다음 구현.**

```
M0. 스캐폴딩       Spring Boot + Testcontainers(Postgres) + Flyway + Swagger
                  → "실 DB로 도는 빈 테스트" 1개 그린부터
M1. 도메인 골격     order/payment/shipping/settlement 4 컨텍스트 패키지 분리
                  → aggregate·불변식을 타입·private 생성자로 박제(DDD)
                  → DDL을 Flyway 마이그레이션으로(CHECK/UNIQUE/EXCLUDE 포함)
M2. 주문→결제       S1·S2 구현, confirm 멱등 + outbox INSERT(같은 TX)
                  → 멱등 동시성 테스트(N스레드 따닥) 먼저 그린
M3. 이벤트→배송     outbox 릴레이 + 배송 구독, source_event_id 멱등
                  → 결정론 시뮬: 이벤트 순서 역전·중복 주입 테스트
M4. 정산 배치       settlement_cycle/line, 멀티셀러 수수료
                  → property: "Σ셀러정산 + 수수료 == 총액" 1만 시나리오
M5. 부분환불       refund + 음수 정산 라인
                  → property: "0 ≤ 누적환불 ≤ 결제" 랜덤 조합
M6. 4-way 대사     recon 배치 + drift 메트릭 + 알람
                  → 독립 경로 재계산, 일부러 1원 깨뜨려 break 검출 테스트
M7. 마감/데모       .http 시나리오, README, Swagger 정리, debug 스냅샷
```

> 각 마일스톤은 **"테스트 그린 + DB 불변식 위반 0 + (해당 시) 대사 drift 0"** 으로 완료를 정의한다. "그럴듯해 보임"으로 완료 판정하지 않는다(§13-6 6번).

## 6. README.md 골격 — "내놓을 때" 이걸 채운다

사이드 프로젝트를 내놓는 README는 **"이게 무엇이고, 어떤 어려운 문제를 어떻게 풀었으며, 어떻게 직접 돌려보나"** 가 전부다. 골격:

```markdown
# 마켓플레이스 커머스 — 멀티셀러 정산·대사 백엔드

## 한 줄 소개
2개 셀러 주문이 결제→배송→정산→환불→4-way 대사까지 흐르는 한 줄기를
end-to-end로 구현. 분산 정합성·동시성·AI 시대 테스트 전략을 코드로 증명.

## 무엇을 증명하나 (4가지)   ← 본 문서 §1 표를 그대로
①멀티셀러 정산+4-way 대사 ②분산 정합성 ③테스트 전략 ④동시성·스케일

## 아키텍처 한 장          ← §3 시퀀스 다이어그램 + 컨텍스트 맵(파일 12 §1)

## 핵심 설계 결정 (자랑 포인트)
- 왜 dual-write를 outbox로 풀었나 (파일 12 §5)
- 왜 정산을 부호 amount 원장으로 (파일 12 §4-2b)
- 왜 4-way 대사인가 (파일 12 §4-4)
- 왜 DB·도메인은 mock 안 하나 (파일 12 §13-3)

## 직접 돌려보기 (Getting Started)
1. docker compose up  (Postgres)
2. ./gradlew bootRun
3. Swagger: http://localhost:8080/swagger-ui
4. 골든 시나리오 실행: scenario.http 를 위에서 아래로  ← §3 S1~S8

## 골든 시나리오 (클릭 흐름)   ← §4 + §3, 각 단계 스크린샷/응답 예시

## 테스트 전략               ← 파일 12 §13 요약 + 실제 테스트 파일 링크
- property(jqwik), 동시성(CountDownLatch), 결정론 시뮬(seed), 대사 drift

## 규모와 진화               ← §7: knee 그래프 + 병목→분리 매핑 + 첫 추출 before/after
- Baseline/Target/Stretch TPS, 발견한 병목, "왜 마이크로서비스인가"를 데이터로

## 한계와 다음 (Scope Out 명시)  ← §2 ❌표: "의도적으로 안 만든 것"과 이유
```

> **README의 차별화는 "핵심 설계 결정"과 "Scope Out"** 이다. 무엇을 왜 그렇게 했고 무엇을 의도적으로 안 했는지가 시니어 신호. 기능 나열이 아니라 *판단*을 보여준다.

## 7. 규모 목표·부하 테스트·마이크로서비스 진화

> 기능(§2~6)이 "무엇을 만드나"라면, 이 절은 **"어느 규모까지, 어떻게 검증하고, 언제·왜 쪼개나"** 다.
> 두 종류의 테스트가 있다 — **정확성 테스트(파일 12 §13: property·대사·DB 불변식)** 와 **성능 테스트(이 절: 부하·병목·한계)**. 정확성이 먼저고, 성능은 그 위에서 측정한다.

### 7-0. 철학 — "측정이 분리를 정당화한다" (조기 분리 금지)

```
   나쁜 길:  처음부터 마이크로서비스로 쪼갬 → 네트워크·분산TX·운영비용을 *근거 없이* 떠안음
   좋은 길:  모놀리스로 시작 → 로컬에서 깨질 때까지 몰아붙임 → 병목을 *데이터로* 발견
            → 그 병목이 "이 서비스를 왜 분리해야 하는지"의 근거가 됨 → 측정 기반 분리
```

핵심은 파일 12 §1과 같다 — **경계는 코드/이벤트로 이미 나눠두되(modular monolith), 물리적 분리(프로세스·DB·배포)는 측정이 정당화할 때만** 한다. 그래서 이 프로젝트는 "왜 마이크로서비스인가"에 *추측*이 아니라 **부하 그래프와 병목 프로파일**로 답한다. 그게 시니어 신호다.

### 7-1. 규모 목표 (TPS 티어)

목표는 "큰 숫자"가 아니라 **"한계를 알고, 한계의 이유를 아는 것"** 이다. 3티어로 잡는다.

| 티어 | 정의 | 주문·결제 쓰기 경로 | 정산/조회 읽기 경로 | 합격 기준 |
|---|---|---|---|---|
| **Baseline** | 손 안 댄 현재 측정 | 측정값 기록 | 측정값 기록 | "지금 몇 TPS인지 안다" |
| **Target** | 로컬에서 무리 없이 | ~300 TPS @ p99<300ms | ~2000 TPS @ p99<100ms | 에러율<0.1%, 정합 유지 |
| **Stretch(한계)** | **일부러 깨뜨림** | knee 지점까지 밀어붙임 | knee까지 | **무엇이 먼저 무너지나 문서화** |

> 측정 지표 4종(USE/RED): **처리량(TPS)·지연(p50/p95/p99)·에러율·자원 포화도(CPU·DB커넥션·락 대기)**. 평균이 아니라 **p99**로 본다 — 평균은 거짓말을 한다.

> **Stretch가 이 절의 진짜 목표.** Target 달성보다 "300을 넘기면 *어디가* 먼저 포화되는가(DB 커넥션? 핫로우 락? outbox 릴레이?)"를 아는 게 마이크로서비스 근거가 된다.

### 7-2. 부하 테스트 — 어떻게

<details>
<summary>🔬 <b>부하 테스트 설계 — 도구·시나리오·측정 환경 통제(결정성)</b></summary>

- **도구**: k6 또는 Gatling(시나리오를 코드로, CI 통합). 골든 시나리오 S1~S8을 **가상 유저(VU)** 스크립트로.
- **부하 패턴 3종**:
  - *Load* — Target TPS를 일정 시간 유지, p99·에러율 합격 확인
  - *Stress* — VU를 계단식으로 올려 **knee(처리량이 안 오르고 지연만 폭증하는 지점)** 탐색
  - *Soak* — Target을 수 시간 유지, **메모리 누수·커넥션 누수·정산 미아** 등 장기 결함 관찰
- **워크로드 분리 측정**: 쓰기 경로(주문·결제)와 읽기 경로(정산 조회)를 **따로** 부하 — 병목 원인이 섞이지 않게.
- **측정 환경 통제(결정성)**: 같은 시드 데이터, 워밍업 후 측정, 백그라운드 잡(정산·대사 배치) ON/OFF 두 조건 비교 → "배치가 OLTP를 잡아먹나"를 본다.
- **관측**: Micrometer→Prometheus→Grafana로 TPS·p99·DB 커넥션 풀 사용률·락 대기·GC를 **부하와 동시에** 그린다. 병목은 그래프에서 *어느 선이 먼저 천장을 치는지*로 보인다.

</details>

### 7-3. 로컬 macOS 한계까지 몰아붙이기 — 어디가 먼저 깨지나

로컬(Apple Silicon, Docker Postgres)은 단일 머신이라 **금방 깨진다 — 그게 좋은 것.** 깨지는 순서가 곧 병목 우선순위다. 예상되는 무너지는 지점:

```
부하 ↑ 올릴수록 순서대로 천장을 친다(예상):
 1) DB 커넥션 풀 고갈      ── HikariCP 한도 < 동시 요청 → 대기 큐 폭증, p99 ↑
 2) 핫로우 락 경합         ── 인기상품 재고 차감/정산 사이클 같은 행에 쓰기 직렬화
 3) 단일 outbox 릴레이 처리량 ── 이벤트 적체, 배송 생성 지연(소비 lag)
 4) 정산/대사 배치 ↔ OLTP 자원 경합 ── 배치 돌면 주문 p99가 같이 튐
 5) PG mock 호출 지연이 스레드 점유 ── 동기 블로킹이면 톰캣 스레드 고갈
 6) CPU/GC                ── 위를 다 넘기면 최후
```

> 각 지점을 **프로파일링(async-profiler, pg_stat_activity, 락 뷰)으로 원인까지** 짚는다. "느리다"가 아니라 **"무엇 때문에 느린지"** 가 산출물.

### 7-3b. 실측 프로토콜 — "예상"을 "측정값"으로 바꾸는 실험 설계

> ⚠️ **이 프로젝트의 1순위 과제.** §7-3은 전부 *예상*이다. 면접에서 "예상 말고 측정값은?"이 들어오면 그 즉시 무너진다(모의 면접 영역 5에서 실증됨). **딱 2개 병목(핫로우 락 · outbox lag)만 실제로 깨뜨려 그래프를 남기면**, "운영 경험 없음"이 "한계를 데이터로 보는 사람"으로 격상된다. 숫자 크기가 아니라 **재현 가능한 실험을 설계·실행·해석할 줄 안다**는 게 증거다.

**실험 1 — 핫로우 락 경합 (재고 차감 직렬화)**

```
가설:    인기상품 1개 행에 동시 쓰기가 몰리면 조건부 UPDATE가 직렬화되어
         TPS는 천장을 치고 p99만 폭증한다(knee).
셋업:    상품 1개 재고 충분, k6로 동일 상품 주문을 VU 1→50→100→200 계단식(Stress)
측정:    ① 처리량(TPS) ② p50/p99 ③ pg_stat_activity의 lock wait 비율
         ④ pg row-level lock(pg_locks) 대기 큐 길이 ⑤ HikariCP active/pending
판정:    TPS가 평평해지는데 p99가 꺾여 올라가는 VU 지점 = knee. 그 순간
         pg_locks 대기가 급증하면 "원인 = 핫로우 락"으로 *국소화* 확정.
대조군:  상품을 100개로 분산(서로 다른 행) → 같은 부하에서 knee가 사라지면
         "병목은 동시성 총량이 아니라 *단일 행 경합*"이 증명됨. ← 이 대조가 핵심
```

**실험 2 — outbox 릴레이 소비 lag**

```
가설:    단일 릴레이의 발행 처리량 < 결제확정 유입 → outbox 적체, 배송 생성 지연.
셋업:    결제확정을 Target 이상으로 쏟고, 릴레이는 기본(단일·배치크기 N)
측정:    ① outbox 미발행 행 수(adt: SELECT count(*) WHERE published_at IS NULL)
         ② 결제확정 → 배송 생성까지의 end-to-end lag(이벤트 타임스탬프 차)
         ③ 릴레이 1회 사이클 처리 건수·소요시간
판정:    유입 > 소비면 미발행 행이 우상향(적체) → "단일 릴레이 처리량 한계" 확정.
대조군:  배치크기↑ 또는 릴레이 2개(파티션 분할) → 적체 곡선이 평탄해지면
         "이벤트 발행 분리/파티셔닝이 정당하다"의 before/after 증거.
```

**측정 결과 표 (실행 후 채운다 — 비어 있으면 §7-4는 여전히 "예상")**

| 실험 | knee 지점 | 그때 p99 | 국소화된 원인(프로파일 근거) | 대조군 효과 |
|---|---|---|---|---|
| ① 핫로우 락 | __ VU / __ TPS | __ ms | pg_locks 대기 __% | 분산 시 knee 소멸 여부 |
| ② outbox lag | 유입 __ TPS | lag __ s | 미발행 행 우상향 기울기 | 파티션 분할 시 평탄화 |

> **해석이 산출물이다.** "300 TPS에서 느려졌다"가 아니라 **"300에서 TPS가 평평해지고 p99가 꺾였으며, 그 순간 pg_locks 대기가 80%였고, 행을 분산하자 knee가 사라졌다 → 원인은 단일 행 경합이며 이것이 재고 서비스 분리(§7-4)를 정당화한다"** 까지가 한 문장으로 나와야 한다. 이 한 문장이 면접에서 영역 5를 ❌에서 ✅로 바꾼다.

### 7-4. 병목 → 마이크로서비스 분리 근거 매핑 (이 절의 핵심 표)

7-3에서 발견한 병목 각각이 **"어떤 분리를, 왜"** 정당화하는지로 직결된다. 이게 면접에서 "왜 마이크로서비스?"에 데이터로 답하는 자료다.

| 발견된 병목(증상) | 근본 원인 | 정당화되는 진화 | 분리 후 기대 |
|---|---|---|---|
| 정산 배치 돌면 주문 p99 폭증 | OLTP와 배치가 같은 DB 자원 경합 | **정산 서비스 분리** + read replica/별도 DB | 배치가 주문을 못 건드림 |
| 인기상품 재고 핫로우 직렬화 | 단일 행 쓰기 경합 | **재고 서비스 분리** + 낙관락/예약카운터/Redis | 재고만 독립 스케일 |
| PG 호출 지연이 톰캣 스레드 점유 | 동기 블로킹 외부 I/O | **결제 서비스 분리** + 비동기/논블로킹 | 외부 지연이 전파 안 됨 |
| outbox 소비 lag, 배송 생성 지연 | 단일 릴레이 처리량 한계 | **이벤트 발행 분리** + 파티셔닝/Kafka | 발행/소비 독립 스케일 |
| 조회가 쓰기 자원을 잡아먹음 | 읽기·쓰기 동일 경로 | **CQRS 읽기 모델** + read replica | 조회 무한 수평 확장 |

> **핵심**: 위 표의 왼쪽(병목)을 *실측*했기에 오른쪽(분리)이 추측이 아니다. **"경계는 이미 나뉘어 있어(파일 12 §1) 분리가 싸다 — 측정이 시점을 정한다."**

### 7-5. 실제 규모로 옮기기 — 진화 경로 + 로컬에서 어디까지 증명하나

```
[1] Modular Monolith (지금)   1프로세스·1DB, 경계는 패키지+이벤트로 분리
        │  ← 7-3에서 병목 실측
[2] 첫 서비스 추출            7-4 표에서 가장 아픈 1개(정산 or 재고)를 별 프로세스+별 DB로
        │  이벤트는 이미 outbox라 그대로, 동기 호출만 끊으면 됨
[3] DB per service           공유 DB → 서비스별 DB, 조인 금지(파일 12 §1)는 이미 지킴
        │
[4] 실 인프라 (설계)          k8s·관리형 Kafka·read replica·오토스케일 — 클라우드 규모
```

**로컬에서 어디까지 *실제로* 증명 가능한가 (정직하게):**

- ✅ **[1] 한계 실측** — 로컬에서 knee와 병목 프로파일까지 진짜 측정 가능.
- ✅ **[2] 한 서비스 추출 후 재측정** — 정산을 별 프로세스+별 DB로 떼고 docker-compose로 같은 머신에서 띄워 **"분리가 병목을 푸나"를 before/after 그래프로** 증명 가능(스케일 자체는 못 키워도 *이음새(seam)가 동작함*을 증명).
- 📐 **[3]~[4]는 설계로** — 진짜 10k TPS 멀티노드는 로컬 불가. 대신 **"7-4 병목 데이터 → 이 인프라가 왜 필요한지"** 를 근거와 함께 설계 문서로. (포트폴리오에선 이 "측정→설계" 연결이 핵심 자랑거리.)

> **시니어 운영 관점(결론):** 사이드 프로젝트가 진짜 차별화되는 지점은 "마이크로서비스로 만들었다"가 아니라 **"모놀리스를 한계까지 밀어 병목을 데이터로 찾고, 그 데이터로 분리 시점·대상·기대효과를 정한 뒤, 첫 서비스 추출로 이음새가 동작함을 실측으로 증명했다"** 는 *서사*다. 추측 없는 진화 — 그게 마스터의 일이다.

## 8. 한 장 요약 (백지 복원용)

```
범위: 돈 한 줄기(주문→결제→배송→멀티셀러 정산→부분환불→4-way 대사)만 깊게.
      쿠폰·교환·프론트·인증은 의도적 Scope Out.
증명: ①정산+대사 ②분산 정합성(outbox·멱등) ③§13 테스트 전략 ④동시성·스케일 —
      하나의 골든 시나리오가 4개를 전부 통과하게 설계.
데모: UI 대신 Swagger + .http 시나리오 + debug 스냅샷으로 "액션→상태변화"를 본다.
순서: 불변식·테스트 먼저(DDD/TDD) → M2 결제멱등 → M3 이벤트 → M4 정산 → M5 환불 → M6 대사.
완료 정의: 테스트 그린 + DB 불변식 위반 0 + 대사 drift 0. "그럴듯함"으로 판정 금지.
규모: 측정이 분리를 정당화. Baseline 측정 → Target(쓰기 300TPS@p99<300ms) → Stretch(일부러 깨뜨려 병목 발견).
      병목(DB커넥션·핫로우락·outbox lag·배치경합) → 각자 어떤 서비스 분리를 정당화하는지 매핑.
      로컬: [1]한계 실측 + [2]한 서비스 추출 before/after 실측, [3~4]는 측정 기반 설계.
README: 한 줄 소개 / 4역량 / 아키텍처 / 핵심 설계 결정 / 돌려보기 / 골든 시나리오 / 테스트 / 규모·진화 / Scope Out.
```

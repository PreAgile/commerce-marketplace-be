Settlement · M4-c

정산 마감, 테이블과 흐름으로 이해하기

멀티셀러 마켓에서 “누구에게 얼마 줄지”를 부호 있는 원장에 쌓고, 주기 단위 봉투(cycle)에 담아 확정하는 구조. 이번 작업이 만든 “마감(close)”과 “조회”를 실제 숫자로 따라갑니다.

01

정산이란 — 돈의 주인이 갈린다

구매자가 한 주문에 셀러 A(1,000원)와 셀러 B(2,000원) 상품을 담아 3,000원을 결제하면, 그 돈은 플랫폼이 일단 다 받습니다. 하지만 대부분 셀러 몫이라 나중에 (매출 − 수수료)를 셀러별로 계산해 지급합니다. 이 계산·확정이 정산입니다.

₩3,000결제 · 플랫폼 일괄 수령
셀러 A매출 1,000 − 수수료 10% (100)₩900
셀러 B매출 2,000 − 수수료 15% (300)₩1,700
플랫폼수수료 100 + 300₩400

실무에서는 배송 한 건마다 즉시 송금하지 않고 주·월 단위로 모아서 지급합니다. 그 주기를 끊고 안의 돈을 확정하는 것이 마감(close) — 이번 작업의 핵심입니다.

02

테이블 두 개 — 원장과 봉투

settlement_line · 원장(ledger)

돈 사실을 부호 있는 줄로 한 개씩 쌓습니다. 매출 SALE + 수수료 COMMISSION −

sellerentry_typeamountcycle_id
ASALE+1,000NULL
ACOMMISSION−100NULL

잔액 컬럼이 없습니다. 셀러 순지급은 SUM(amount_minor)로 도출 → 여기선 1,000 + (−100) = 900. 정정도 UPDATE가 아니라 반대 부호 줄을 추가합니다(회계 장부 원리). 부호 규칙은 코드가 아니라 DB CHECK가 강제합니다.

settlement_cycle · 봉투(주기)

셀러별 정산 주기를 나타내는 껍데기. 기간 [start, end) 과 상태를 가집니다.

cycle_idsellertypeperiodstatus
1Aweekly06-01 ~ 06-08CLOSED

“셀러 A의 6월 1주차 봉투”. 마감하면 원장 줄들이 이 봉투에 담깁니다(줄의 cycle_id가 이 봉투를 가리킴). 기간 겹침 금지(EXCLUDE) 제약이 같은 셀러·유형의 겹치는 봉투를 DB에서 거부 → 매출 이중 집계 차단.

마감은 금액을 단 1원도 만들거나 옮기지 않습니다. 오직 원장 줄의 cycle_id를 NULL → 1 로 채워 “소속을 확정”할 뿐입니다.

05

마감 로직 & “열림/닫힘”의 진짜 의미

세 걸음 (SettlementCloseService.run)

① 대상 셀러cycle_id IS NULL AND eligible_at < end 인 줄이 있는 셀러들 (락 순서 ORDER BY seller_id)
② 봉투 + 담기셀러별 settlement_cycle INSERT(바로 CLOSED) + UPDATE ... SET cycle_id WHERE cycle_id IS NULL
③ 요약봉투별 순지급 SUM 을 응답으로 반환

eager 대신 deferred 를 택했다

봉투를 미리 OPEN으로 열어두지 않습니다. 줄은 cycle_id=NULL로 그냥 떠 있고(=사실상 “열린” 상태), 마감이 그 NULL 줄들을 쓸어 담아 바로 CLOSED 봉투를 만듭니다.

왜? 배송완료 핫패스는 “줄 하나 넣기”만 하면 되고, “언제·어떤 주기로 끊을지”는 마감 배치로 분리됩니다. 덤으로 유입과 마감이 같은 봉투를 두고 경합할 일이 사라집니다.

cutoff 스윕 — 늦게 온 줄을 안 잃어버린다

대상 조건에 하한(eligible_at >= start)을 두면, 택배사 콜백 지연으로 마감 후 도착한 줄이 이미 닫힌 봉투(재마감 불가)에 못 들어가 영구 고립됩니다. 그래서 하한을 없애고 eligible_at < end 만 봅니다(cutoff). 늦은 줄은 NULL로 남아 다음 마감이 자연히 흡수 — 정산이 한 주기 밀릴 뿐 돈은 안 잃습니다.

06

동시성 방어 — 코드 if가 아니라 DB 구조로

돈이라 “동시에 두 번 마감해도 이중 지급 없음”이 필수. 세 겹으로 막습니다.

① 봉투 이중 생성

EXCLUDE (GiST)

같은 셀러·유형·겹치는 기간의 두 번째 봉투 INSERT를 DB가 거부. 재실행·동시 실행 모두 봉투는 1개. 배치 전체 롤백이라 부분 마감도 없음.

② 줄 이중 귀속

WHERE cycle_id IS NULL

한 마감이 줄을 담으면(cycle_id 채워짐) 다른 마감의 UPDATE ... WHERE cycle_id IS NULL은 그 줄을 더 못 건드림. 조건부 UPDATE의 배타락이 직렬화.

③ 고스트 봉투 리뷰 적중

assigned == 0 → DELETE

EXCLUDE 키에 cycle_type이 있어 유형이 다르면(weekly·express) 안 막힘 → 늦은 쪽이 줄 0개 빈 봉투를 남김. 담긴 줄 0이면 방금 만든 봉투를 되돌림.

③은 CodeRabbit이 Major로 지적한 실제 버그. 가드를 뺀 채 8스레드 크로스-타입 동시 마감으로 8/8 재현해 반례를 테스트로 박제한 뒤 고쳤습니다.

07

조회 — 셀러가 자기 정산을 본다

GET /sellers/A/settlements

// 마감분(봉투별) + 아직 안 담긴 pending
{
  "sellerId": 10,
  "cycles": [
    { "cycleId": 1, "status": "CLOSED",
      "lineCount": 2, "netMinor": 850 }
  ],
  "pendingNetMinor": 360,
  "totalNetMinor": 1210
}

세 값 모두 SUM 도출

cycles[].netMinor마감된 봉투별 순지급 = 그 봉투 줄들의 SUM
pendingNetMinor아직 cycle_id=NULL 인 줄들의 합 = “열려 있는” 돈
totalNetMinor850 + 360 = 1,210 (전체 원장 순액)

두 값을 한 스냅샷(REPEATABLE READ) 에서 읽어, 조회 도중 마감이 끼어들어 “봉투엔 담겼는데 pending에도 세는” 모순을 막습니다.

↹

한 줄 요약. 정산 = 셀러별 (매출 − 수수료)를 부호 원장에 줄로 쌓고, 주기 단위 봉투(cycle)에 담아 확정하는 것. 이번 작업은 그 “담아 확정(마감)”과 “조회”를 만들었고 — 돈이라 이중 마감·이중 집계·고스트 봉투를 EXCLUDE·조건부 UPDATE·가드로 구조적으로 봉쇄한 게 핵심입니다.

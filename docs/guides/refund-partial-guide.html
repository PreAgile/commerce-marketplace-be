Settlement · M5 · 설계 지도(계획)

부분환불 — 이미 잡힌 돈을 되돌리는 길

구매자가 주문 라인 일부를 환불받으면, 셀러 몫으로 잡혔던 매출을 음수 원장 줄로 되돌립니다. payment가 0 ≤ 누적환불 ≤ 결제 불변식을 지키고, settlement가 이벤트로 REFUND 줄을 기록합니다. 아직 구현 전 — 이 문서는 무엇을 어떻게 만들지의 지도입니다.

이미 있음 (DDL·제약) M5에서 만들 것 당신이 정할 갈림길
01

문제 — 매출은 이미 셀러 몫으로 잡혔다

M4까지 흐름이 끝나면 셀러 A의 매출은 이미 원장에 줄로 쌓여 있고(마감됐을 수도), 곧 지급 대상입니다. 그런데 구매자가 그 라인의 절반(500원)을 환불받으면? 이미 "줄 돈"으로 잡힌 매출을 되돌리고, 떼갔던 수수료도 일부 돌려줘야 합니다.

환불 요청 M5 신규

POST /orders/{id}/refunds { line, amount: 500 } — 주문 라인 지정 + 환불 금액. (spec S6 · §4 API 표)

02

핵심 개념 — 환불도 "음수 줄"이다 (수정·삭제 아님)

부호 원장의 원리 그대로입니다. 매출 줄을 고치거나 지우지 않고, 반대 방향 줄을 새로 추가합니다. 환불 하나에 두 줄이 생깁니다 — 매출 되돌림과 수수료 환급.

▸ 환불 전 — 셀러 A 원장 (net 900)
entry_typeamount의미
SALE+1,000매출
COMMISSION−100수수료 10%

SUM = 900 (셀러가 받을 돈)

↓  환불 500 (라인의 절반)  ↓
▸ 환불 후 — 두 줄 추가 (net 450)
entry_typeamount의미
SALE+1,000그대로
COMMISSION−100그대로
REFUND−500매출 되돌림 (음수)
REFUND_COMMISSION+50수수료 환급 (양수)

SUM = 1000 − 100 − 500 + 50 = 450. 절반 환불했으니 셀러도 절반만 가짐(500 − 50).

이미 있음 settlement_line의 REFUND(−)·REFUND_COMMISSION(+) entry_type과 부호 CHECK는 V5 DDL에 이미 정의돼 있습니다. M5는 이 줄을 채우는 슬라이스입니다.

03

흐름 — 두 BC를 이벤트로 잇는다

환불은 두 컨텍스트를 지납니다. 직접 참조 대신 기존 트랜잭셔널 아웃박스 패턴(결제→배송, 배송→정산과 동일)으로 잇습니다.

1

환불 요청 접수 order/web 신규

POST /orders/{id}/refunds {line, amount} — 어느 라인을 얼마 환불할지 받는다.

2

PG 환불 + 누적환불 반영 payment BC 갈림길

PG mock으로 환불 요청 → 누적환불액 증가. 불변식 0 ≤ 누적환불 ≤ 결제의 주인은 payment. 저장 방식이 §05의 갈림길.

3

PaymentRefunded 이벤트 적재 payment BC 신규

환불 반영과 같은 트랜잭션으로 outbox에 이벤트를 쌓는다(dual-write 원자화, ADR-002/013). 커밋돼야 발행된다.

4

정산 줄 append settlement BC 신규

이벤트 소비 → REFUND REFUND_COMMISSION 두 줄을 cycle_id=NULL로 append. 멱등키 UNIQUE로 재전달 흡수(ADR-010 복합키).

왜 이벤트인가: payment가 죽어도 정산이 자기 페이스로 소비하고, 나중에 settlement를 분리 추출해도 경계가 안 깨진다. M3·M4가 쓴 그 패턴을 그대로 재사용 — 새 인프라 0.

04

이미 마감된 매출을 환불하면? — cutoff가 흡수

M4-c의 cutoff 스윕을 그대로 탄다

6월 매출이 이미 CLOSED 사이클에 묶였는데 7월에 환불이 오면? 닫힌 사이클을 다시 열지 않는다. REFUND 줄을 cycle_id=NULL로 넣으면 다음 마감이 흡수해 그 주기 정산액에서 차감된다.

= "6월 매출을 7월에 환불하면 7월 정산에서 −450". 실서비스 정산과 동일한 late-arrival 처리.

▸ 환불 줄 (미배정 → 다음 마감)
entry_typeamountcycle_id
REFUND−500NULL
REFUND_COMMISSION+50NULL

M4-c에서 만든 마감 로직이 수정 없이 이 줄들을 다음 주기에 담는다.

05

갈림길 — 당신이 정할 핵심 결정

누적환불을 refunded_amount 컬럼에 둘까, append-only 원장으로 옮길까? DECISION

payment(V2)엔 지금 refunded_amount 컬럼과 CHECK (0 ≤ refunded_amount ≤ paid_amount)가 이미 있습니다. 그런데 V2 주석이 예고합니다 — "환불을 append-only 원장으로 옮기고 누적환불은 SUM으로 도출 예정, 잔액 컬럼 UPDATE 금지와 정합시키려." 여기가 M5의 설계 핵심입니다.

A. refunded_amount 컬럼 유지

  • 단순 — 환불마다 컬럼 UPDATE
  • CHECK가 상한을 이미 강제
  • 잔액 컬럼 UPDATE = AGENTS.md §1.2 황금률과 충돌
  • 동시 환불 시 lost update 위험(락 필요)

B. refund_ledger append-only 권장 방향

  • 황금률 정합 — 누적환불 = SUM 도출
  • 정산 원장과 같은 철학(부호·append-only)
  • 멱등키 UNIQUE로 따닥 환불 방어
  • 새 테이블 + 상한 검증을 어디서(트리거/앱)

V2 TODO·ADR-010이 이미 B 방향을 가리키지만, 최종 결정과 상한 강제 위치(payment 트리거 vs 앱)는 당신이 정합니다. 정하면 ADR로 박제.

06

통과할 것 — 불변식과 테스트 (완료의 정의)

불변식 (돈 안전)

상한0 ≤ 누적환불 ≤ 결제금액 — payment가 강제(CHECK 또는 원장 SUM)
보존환불 후 셀러 net = 매출 − 수수료 − 환불 + 환불수수료, 음수 아님
멱등같은 환불 재전달 → 줄 한 벌만 (source_event_id UNIQUE)

테스트 (spec M5)

  • ◇property: 0 ≤ 누적환불 ≤ 결제 랜덤 조합 폭격 (부분·전액·초과 시도)
  • ◇예제 IT: 부분환불 → REFUND/REFUND_COMMISSION 줄 생성·부호·금액
  • ◇경계: 결제액 초과 환불 거부(CHECK), 0원 환불
  • ◇동시성: 따닥 환불 2스레드 → 초과 안 됨
  • ◇멱등: 이벤트 재소비 → 줄 한 벌

한 줄 요약. 부분환불 = 이미 잡힌 매출을 REFUND(−)·REFUND_COMMISSION(+) 음수 줄로 되돌리는 것. payment BC가 0≤누적환불≤결제를 지키고, settlement가 이벤트로 줄을 기록하며(M3·M4 아웃박스 재사용), 이미 마감된 매출의 환불은 M4-c cutoff가 다음 주기로 흡수. 핵심 갈림길은 누적환불을 컬럼으로 둘지 append-only 원장으로 옮길지.

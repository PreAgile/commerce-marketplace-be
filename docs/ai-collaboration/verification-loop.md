# 검증 루프 — AI 코드를 "그럴듯함"이 아니라 "증거"로 믿는다

> 방법론([`methodology.md`](./methodology.md))을 *실제 작업 절차*로 옮긴 것. 이 프로젝트에서 AI 산출물을 신뢰하는 기준입니다.

## 신뢰 기준 — "완료"의 정의

AI가 "다 됐습니다"라고 해도, 다음 3개가 모두 초록일 때만 완료로 인정합니다:

```
✅ 테스트 그린            (단위·통합·property·동시성·결정론 시뮬)
✅ DB 불변식 위반 0        (CHECK/UNIQUE/EXCLUDE가 거부한 적 없음 = 데이터 정합)
✅ 대사 drift 0           (독립 경로 재계산이 일치)
```

**"코드가 그럴듯해 보임"은 완료 기준이 아닙니다.** 그럴듯함은 AI가 가장 잘하는 거짓말이기 때문입니다.

## 작업 사이클

```
1. [명세] 사람이 불변식·테스트(채점 기준)를 먼저 확정/검수
2. [생성] AI에게 "이 테스트를 통과시켜라" (Bounded Context 1개 단위)
3. [검증] 테스트 그린 + DB 불변식 + 대사 → 셋 다 초록인가?
4. [반례] 초록이어도 property/시뮬로 "안 생각한 입력·순서"를 폭격
5. [정정] 깨지면 반례를 명세로 추가하고 2로 — 반례가 영구 회귀 테스트가 됨
```

## 이 프로젝트에서 실제로 한 AI 활용 (정직한 기록)

설계 단계에서 AI를 단순 코드 생성기가 아니라 **적대적 검증 파트너**로 썼습니다:

1. **설계** — DDD 경계·DDL·불변식을 AI와 함께 도출([`../design/system-design.md`](../design/system-design.md)).
2. **자기 적대 검증** — AI를 *대기업 시니어 면접관* 역할로 세워, 자기 설계를 꼬리질문 5~7단계로 **공격**하게 했습니다.
3. **오류 발견** — 그 과정에서 설계의 실제 오류를 잡았습니다. 예:
   - "마감 vs 유입을 `FOR SHARE`로 직렬화" → **틀림**(공유 락은 상호배제 안 됨). 유입=`FOR SHARE`/마감=`FOR UPDATE` **비대칭**으로 정정.
   - "drift≠0 = 알람" → **틀림**(PG 입금 시차로 정상 drift 상존). recon 상태 모델(EXPECTED/IN_TRANSIT/SETTLED/BREAK)로 승격.
   - "정산계도 순차 처리로 순서 보장" → **불필요**(부호 원장 SUM은 교환법칙적). 멱등만으로 병렬 처리로 정정.
4. **정정 반영** — 발견한 오류를 설계 문서에 다시 박제.

> 이 사이클 자체가 방법론의 증거입니다 — **AI가 만든 것을 AI로 적대적으로 검증하고, 사람이 최종 판단**합니다.
> 결정별 상세는 [`decision-log.md`](./decision-log.md)에 ADR로 기록합니다.

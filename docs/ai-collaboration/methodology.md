# AI 협업 방법론 — "AI가 못 어기게, 검증 가능하게"

> 이 프로젝트는 AI(Claude)를 적극 활용해 설계·구현했습니다. 그러나 "AI에게 맡겼다"가 아니라,
> **AI가 넘을 수 없는 선을 구조로 박제하고, 검증 가능한 명세로 채점**하는 방식을 실천했습니다.
> 이 문서는 그 방법론이고, 실제 검증 루프는 [`verification-loop.md`](./verification-loop.md), 결정 기록은 [`decision-log.md`](./decision-log.md)에 있습니다.

## 핵심 명제

> AI 코드엔 "사람이 모르는 버그"가 더 많다. 그래서 검증을 **사람의 코드 리뷰("그럴듯함")** 에 의존하지 않고,
> **구조(타입·DB 제약)와 명세(테스트)와 런타임 안전망(대사·불변식)** 으로 옮긴다.

이 명제는 AI 시대에 *새로 생긴* 게 아니라(토스·배민이 이미 쓰던 3중 방어), AI가 **우회 경로를 더 빨리·더 많이 만들어** 리뷰 커버리지를 구조적으로 무력화하기 때문에 그 가치가 *증폭*됩니다.

## 4개 도구 — AI를 어디에 가두나

### 1. DDD — "AI가 넘으면 안 되는 선"을 코드·타입·DB로 박제
- Bounded Context 1개 = AI 작업 1단위. 남의 내부는 이벤트 페이로드로만 본다.
- 불변식(환불 ≤ 결제, 정산 부호 규칙)을 **private 생성자·값 객체·DB CHECK/UNIQUE/EXCLUDE**로 박제 → AI가 우회 경로를 새로 짜도 못 뚫는다.
- 경계가 미리 나뉘어 있어, 나중에 마이크로서비스로 쪼갤 때 비용이 싸다.

### 2. TDD — "검증 가능한 명세"로 AI에게 채점 기준을 준다
- 돈·정산·멱등·상태머신은 **테스트 먼저**. 테스트 = AI에게 주는 *채점 기준*.
- AI에게 "좋은 코드 짜줘"(모호) 대신 **"이 테스트들을 통과시켜라"**(명세)를 준다.
- 단언(assert)은 사람이 검수한다 — 테스트가 곧 명세이므로 거기에 거짓이 없어야 한다.

### 3. Mock 경계 — DB·도메인은 절대, 외부만
- **mock한 것 = 검증 안 된 것.** 불변식의 절반이 DB 제약에 있으므로 가짜 repo는 거짓 안심을 준다.
- DB·도메인 로직 = **진짜**(Testcontainers 실 Postgres). PG·택배·시계(Clock) = **mock**.
- "상태 검증, `verify(interaction)` 금지"를 AI에게 명시해 mock 남발을 억제한다.

### 4. 테스트가 *구조적으로* 못 보는 것 — 런타임 안전망
예제 테스트는 "내가 생각한 케이스"(점)만 본다. 나머지는:
- **Property-based test** — "법칙"(정산 총액 보존, 0≤누적환불≤결제)을 랜덤 수천 시나리오로 폭격
- **결정론적 시뮬레이션** — 이벤트 순서 역전·동시성·장애를 시드로 통제해 재현
- **런타임 대사(reconciliation)** — 독립 경로로 재계산한 두 숫자를 맞춰 drift를 상시 감시
- **DB 불변식** — 잘못된 데이터는 INSERT 자체가 거부되게(최후 보루)

## 3중 방어선

```
[1] 배포 전   테스트(TDD/property/시뮬)   "알려진·생성된 케이스를 막는다"
[2] INSERT시점 DB 불변식(CHECK/UNIQUE/EXCLUDE) "잘못된 데이터는 들어오는 순간 거부"
[3] 런타임 주기 대사 배치                    "독립 계산을 맞춰 어긋나면 알람"
```

> 결론: **테스트는 "내가 아는 버그"를, DB 불변식은 "모든 경로"를, 대사는 "내가 모르는 버그가 만든 차이"를** 잡는다.
> 상세 설계는 [`../design/system-design.md`](../design/system-design.md) §13, 적용 범위는 [`../design/implementation-spec.md`](../design/implementation-spec.md)를 참조하세요.
